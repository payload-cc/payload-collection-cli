# Payload Collection CLI - Setup Instructions

This document specifies the complete implementation, build, and publish steps for the `payload-collection-cli` package. Note that the instructions herein represent the full reference logic for the CLI project.

## 1. Project Structure

```text
payload-collection-cli/
├── src/
│   ├── types.ts         # Configuration types
│   ├── resolver.ts      # Relation lookup logic
│   ├── executor.ts      # CRUD & JSONLines engine
│   └── bin.ts           # CLI entry point (Auto-discovery)
├── tests/               # Unit testing framework
│   └── executor.test.ts # Core logic tests
├── .github/workflows/   # CI/CD Workflows
├── package.json
├── tsconfig.json
└── tsup.config.ts       # Build configuration
```

## 2. Dependencies Setup

Run the following commands to initialize and install dependencies:

```bash
pnpm init
pnpm add jiti
pnpm add -D typescript tsup payload next react @types/node vitest
```

## 3. Implementation Code

The module is responsible for robust collection management targeting a user's `payload.config.ts`.
- **`src/types.ts`**: Holds mapping configurations and CLI configuration definitions.
- **`src/resolver.ts`**: The core graph relationship resolver logic that intercepts `relationship`-type properties from payloads, looking them up on mapped search indices.
- **`src/executor.ts`**: A high performant abstraction for looping over `jsonl` inputs, dispatching them through the standard `payload.create/update` APIs. 
- **`src/bin.ts`**: The primary executable file. Discovers user configurations (`payload-collection-cli.config.ts`), boots Payload seamlessly over Node.js bindings using `jiti`, and delegates execution.

## 4. Testing

Tests are written using `vitest` to ensure proper functional coverage without bloating the final distributable package via `peerDependencies` mocks.
The `tests/` directory is automatically excluded from the dist output payload per configuration.
Run tests via:
```bash
pnpm test
```

## 5. CI/CD Pipeline

The project uses GitHub Actions for scalable integration processes.
- **Continuous Integration (`.github/workflows/ci.yml`)**: Enforces validation gating; ensuring `pnpm test` and `pnpm build` resolve cleanly on every active pull-request.
- **Automated Deployments (`.github/workflows/publish.yml`)**: Standardizes the npm public registry publication routine. It engages automatically upon GitHub release publication, validating through `NPM_TOKEN` registry secrets.

## 6. Publish Flow

1. **System Build Test:** Manually run `pnpm build` assuring that `dist/bin.js` yields.
2. **Local Link Test:** Execute `pnpm link --global` locally, navigate to a separate Payload 3.0 project folder, and observe execution on `payload-collection-cli collections create '{"title": "test"}'`.
3. **Public Release:** Release tags on GitHub natively invoke production deploy pipelines via Actions workflows.
